# pprof 问题排查实战案例（Demo 规划）

## 目标

通过“故意制造问题 → pprof 定位 → 优化 → 对比验证”的闭环，训练线上排查能力与面试表达。

## Demo 1：goroutine 泄漏（协程不退出）

- **现象**：请求结束后 goroutine 数持续上涨，内存/FD 逐步增长
- **采集**：
  - goroutine profile（看堆栈与阻塞点）
  - heap/alloc（看是否伴随分配增长）
- **你要在 pprof 里找到的证据**：
  - 大量 goroutine 堆栈停在同一阻塞位置（chan receive / select / netpoll / wait）
- **优化方向**：
  - 补齐取消（context）、超时、关闭通道、退出条件

## Demo 2：锁竞争（吞吐低 / 延迟高）

- **现象**：CPU 不高但吞吐上不去，P99 上升明显
- **采集**：
  - mutex profile（争用在哪）
  - block profile（等待在哪）
  - CPU profile（辅助看是否自旋/争用开销）
- **证据**：
  - 某把锁的持有/等待占比很高；调用链集中
- **优化方向**：
  - 缩小临界区、读写锁、分片锁、无锁结构、减少共享

## Demo 3：内存泄漏 / 分配过多（GC 压力）

- **现象**：RSS/heap 持续增长或 GC 频繁，尾延迟变差
- **采集**：
  - heap（inuse）看存活对象
  - alloc（alloc_space/alloc_objects）看分配热点
  - CPU（看 GC 相关开销）
- **证据**：
  - 某类对象持续存活，或某路径分配异常集中
- **优化方向**：
  - 降低分配（逃逸分析）、复用（sync.Pool）、减少长生命周期引用

## 你最终要沉淀的“标准输出”

- **问题描述**：现象 + 指标（QPS/P99/heap/goroutines）
- **采样选择**：为什么采这个 profile
- **证据链**：pprof 输出截图/关键函数/调用链
- **根因**：一句话根因 + 原理解释
- **改动**：最小修改点
- **验证**：前后对比数据与结论
