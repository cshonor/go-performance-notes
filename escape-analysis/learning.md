# 逃逸分析（escape-analysis）学习路径（第三优先级）

## 为什么第三个学

- 逃逸分析是**内存优化的基础**，直接影响堆大小与 GC 压力，是 pprof 内存分析的“原理支撑”
- 先学 pprof 再学逃逸分析，能快速建立因果链：**堆分配多 → GC 压力大 → 尾延迟变差**

## 学习路径（闭环）

### 1) 基础使用

- 看逃逸日志：
  - `go build -gcflags="-m -m" ./...`
  - `go test -gcflags="-m -m" ./...`

### 2) 核心规则（吃透常见 10 类场景）

> 目标：看到代码就能猜“可能逃逸的原因”，并用 `-m` 输出验证。

- 指针相关逃逸（生命周期超出栈帧）
- 接口相关逃逸（装箱/多态边界）
- 闭包捕获变量导致逃逸
- 切片/Map 扩容、间接引用导致的分配与逃逸（结合具体代码路径）
- 反射、`fmt` 等让编译器难以证明不逃逸的路径

### 3) 实战优化（用 pprof 验证）

- 先用 alloc/heap 找“分配热点”
- 回到代码用逃逸分析定位“为什么在堆上”
- 做最小改动减少堆分配
- 再用 pprof 对比验证（`allocs/op`、`B/op`、GC 次数/时间等）

## 面试考点（高频）

- 什么情况下变量会逃逸（典型例子要能讲）
- 如何减少逃逸（让编译器看得懂、减少接口边界/闭包/反射等）
- 逃逸分析的作用：为什么能影响 GC 与延迟
