# sync.Pool 使用指南（全维度）

## 定位：它解决什么问题

`sync.Pool` 用于复用**高频、短生命周期、可丢弃可重建**的临时对象，目标是：

- 减少分配次数
- 降低 GC 压力
- 改善吞吐与尾延迟（在分配/GC 是瓶颈时）

## 适用/不适用场景（先把边界立住）

### 适用

- `[]byte`/buffer/临时结构体等
- 每次请求都创建、用完即丢的对象

### 不适用（高频坑）

- 需要“强保证一定存在”的对象（Pool 不是强语义对象池）
- 长生命周期对象、包含外部资源引用的对象
- 大对象（可能导致内存占用抖动），需严格验证

## 基础用法要点

- `Get()`：拿对象（可能返回 nil，需要兜底创建）
- `Put()`：放回池（放回前要 reset，避免残留引用导致泄漏）

## 与 pprof/逃逸分析的联动（标准动作）

1. 用 pprof alloc/heap 找到分配热点
2. 用逃逸分析确认对象确实在堆上、确实值得优化
3. 引入 Pool 复用并做好 reset
4. 用 pprof 重新采集对比（allocs/B、GC、P99）
