# sync.Pool 底层原理（与 P 绑定、GC 清理）

## 你需要掌握的“最小原理集”

- `sync.Pool` 设计目标是“复用临时对象”，不是“强一致对象池”
- 为了高并发性能，Pool 采用**本地缓存**（与 P 关联）减少争用
- GC 时 Pool 可能被清理（对象可丢弃），因此不能依赖其持久性

## 为什么要和 P 绑定（面试必答点）

- 减少跨线程/跨 P 竞争
- 提高局部性与吞吐
- 让 Get/Put 在高并发下仍保持较低开销

## 为什么 GC 会清理 Pool（一定要会讲）

- Pool 里的对象本质上是“可回收的缓存”
- GC 清理能避免缓存无限增长与长期占用内存
- 这也是它不能当“连接池/资源池”的根本原因之一

## 与优化的关系（怎么形成证据链）

- 是否值得用 Pool：取决于“分配/GC 是否是瓶颈”
- 必须用 pprof 验证，而不是凭感觉
