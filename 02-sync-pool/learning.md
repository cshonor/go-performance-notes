# sync.Pool 学习路径（第四优先级）

## 为什么第四个学

- `sync.Pool` 是**内存优化的核心工具**，依赖：
  - 逃逸分析：判断对象是否在堆上、是否值得复用
  - pprof：验证优化是否真实有效（减少分配/降低 GC/改善尾延迟）
- 目标是直接降低 GC 压力，是高并发服务的常见优化手段

## 学习路径（闭环）

### 1) 基础使用

- 掌握 `Get()` / `Put()` 的基本用法
- 理解适用场景：**高频、短生命周期、可丢弃可重建**的临时对象（如 buffer、临时结构体）

### 2) 底层原理（面试必问方向）

- Pool 的本地池（与 P 绑定）与全局池的设计动机
- GC 清理机制：为什么 Pool 不能当“强保证的对象池”

### 3) 实战优化（必须用 pprof 验证）

- 先用 pprof 的 alloc/heap 找到分配热点
- 引入 `sync.Pool` 做复用（注意 reset / 清理引用）
- 再次采集对比：
  - `allocs/op`、`B/op`
  - GC 次数/时间
  - P95/P99 延迟（如果是延迟型业务）

## 面试考点（高频）

- `sync.Pool` 底层实现与关键行为（为什么会被 GC 清空）
- 为什么 Pool 要和 P 绑定（并发效率与局部性）
- 适用场景与反例（什么时候用反而更差）
