# 逃逸分析使用指南（全维度）

## 定位：它解决什么问题

逃逸分析回答的是：**变量最终分配在栈还是堆**，从而指导你减少堆分配、降低 GC 压力。

## 常用命令

- 构建时查看逃逸信息：
  - `go build -gcflags="-m -m" ./...`
- 测试时查看逃逸信息：
  - `go test -gcflags="-m -m" ./...`

## 怎么读输出（最小要点）

- 关注“**escapes to heap**”相关提示
- 关注“因为什么原因逃逸”的解释（接口、闭包、返回地址等）
- 目标是建立映射：**代码结构 → 逃逸原因 → 优化方向**

## 高频逃逸场景（先背下来）

- 返回局部变量地址（生命周期超出栈帧）
- interface 装箱（把具体类型放进接口边界）
- 闭包捕获变量（地址被延长）
- 反射、`fmt`、过度抽象导致编译器难以证明不逃逸
- 切片/Map 扩容等导致分配（可能伴随逃逸）

## 优化手法（先从可控改动开始）

- 减少接口边界与装箱（必要时用泛型/具体类型）
- 避免不必要的闭包捕获
- 让编译器“看得懂”（减少过度抽象/反射）
- 与 `pprof` 联动：alloc 热点 → 逃逸点 → 改动 → 对比验证
