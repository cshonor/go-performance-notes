# 逃逸分析底层原理（编译器视角）

## 你需要掌握的“最小原理集”

- **目的**：决定变量放在栈还是堆，保证引用在生命周期内有效
- **核心约束**：如果编译器无法证明“引用不会在函数返回后继续被使用”，就倾向于让对象**逃逸到堆**
- **影响**：堆分配增加 → GC 压力增大 → 尾延迟可能变差（需用 pprof 验证）

## 栈/堆分配的差异（面试表达版）

- 栈：分配释放快，随栈帧结束自动回收，GC 不管
- 堆：需要 GC 管理，分配与回收成本更高，可能导致更多 GC 周期与 STW/辅助 GC

## 常见规则背后的“为什么”

- 返回地址：返回指针意味着引用可能跨越当前栈帧
- 接口装箱：在接口边界上，编译器需要保守处理（具体逃逸取决于场景与优化）
- 闭包捕获：变量地址被闭包持有，生命周期被延长
- 反射/动态行为：编译器难以静态证明，容易走保守路径

## 与优化的关系（建议记住这句话）

> 逃逸分析不是让你“为不逃逸而不逃逸”，而是让你在**性能瓶颈真实存在**时，用最小改动减少不必要的堆分配。
